> 时间跨度: 2026-05-18 ~ 2026-05-28
> 涉及文件: 20+ 个
> 核心问题: 博客创建/编辑/删除失败，修改后内容不更新

## 一、问题全景

自全动态迁移（D1 + Workers API）完成后，博客模块出现一系列连锁问题：

| 症状 | 出现频率 | 严重度 |
|---|---|---|
| 编辑保存后刷新回到旧内容 | 100% | 🔴 |
| 新建文章 UNIQUE 约束报错 | 经常 | 🔴 |
| 删除返回 401 | 经常 | 🔴 |
| 旧文章无法编辑/删除 | 100% | 🟡 |
| 中文标题文章 GET 404 | 部分 | 🔴 |
| "保存中"遮罩不消失 | 新建后 | 🟡 |
| 每操作都要重新输入密码 | 偶发 | 🟡 |
| Service Worker 缓存 API 404 | 偶发 | 🟡 |

## 二、根因清单及修复

### Bug 1: D1 主从复制延迟导致编辑后内容回退

**现象**: 编辑保存成功（PUT 返回 ok），详情页显示新内容，刷新后变回旧内容。

**根因**: D1 是主从架构。PUT 写主库成功后，Worker 紧接着 SELECT 回读会命中从库——从库可能尚未同步。旧代码:
```javascript
// _worker.js PUT /api/posts/:slug
await env.DB.prepare("UPDATE posts SET ... WHERE slug=?").bind(...).run();
const row = await env.DB.prepare("SELECT * FROM posts WHERE slug = ?").bind(slug).first();
return json({ ok: true, post: row }); // row 可能是旧数据！
```

**修复**: PUT 后绝不 SELECT 回读。前端 Editor 用自身 state 构建返回数据:
```javascript
// Editor.tsx — 用本地 state 构造 savedPost
onSaved?.({ slug: postSlug, title, content, excerpt, tags, created_at: new Date().toISOString() });
```

**日期**: 2026-05-18
**涉及文件**: `_worker.js`, `Editor.tsx`

---

### Bug 2: 前端双重 POST 创建文章

**现象**: 新建文章有时成功，有时 UNIQUE 冲突报错；保存后跳转到列表而非详情。

**根因**: Editor 的 `handleSave` 自己调了一次 `api.savePost`，然后 `onSaved` 回调又触发 BlogShell 的 `handleCreate` 再调一次。同一文章 POST 两次:
```javascript
// Editor.tsx (旧代码)
await api.savePost({ slug: postSlug, ... }); // ← 第一次 POST
onSaved?.({ slug: postSlug, ... });          // → handleCreate → api.savePost(...) ← 第二次 POST
```

第二次 POST 同 slug → UNIQUE 约束冲突。加入 ON CONFLICT 后第一次 POST 成功而第二次 UPDATE，掩盖了双重调用的问题但没解决根本症结。

**修复**: Editor 不再调用 API，只通过 `onSaved` 传数据给 BlogShell 统一处理:
```javascript
// Editor.tsx (新代码)
const postData = { slug: postSlug, title, content, excerpt, tags, created_at: new Date().toISOString() };
onSaved?.(postData); // 仅传数据，不调 API
```

**日期**: 2026-05-28
**涉及文件**: `Editor.tsx`, `BlogShell.tsx`

---

### Bug 3: Worker POST 路由去掉了 ON CONFLICT

**现象**: 创建文章时 UNIQUE 约束报错。

**根因**: 重构 Worker 时为避免"新文章覆盖旧文章"问题，去掉了 `ON CONFLICT(slug) DO UPDATE` 子句。纯 INSERT 遇重复 slug → SQLITE_CONSTRAINT。

**修复**: 加回 `ON CONFLICT`:
```sql
INSERT INTO posts (...) VALUES (...)
ON CONFLICT(slug) DO UPDATE SET
  title=excluded.title, content=excluded.content,
  excerpt=excluded.excerpt, tags=excluded.tags, updated_at=excluded.updated_at
```

**日期**: 2026-05-28
**涉及文件**: `_worker.js`

---

### Bug 4: `ensureAuth` 回调未正确等待

**现象**: 所有写操作返回 401 Unauthorized，但无密码输入框弹出。

**根因**: `ensureAuth` 函数被多次重构，某次重构时将回调类型改为 `() => Promise<void>` 并引入复杂的 Promise 包装。但 `onDone` 调用未 `await`，异步回调在后台执行——API 调用还没发出，`ensureAuth` 就返回了:
```javascript
// useAuth.tsx (中间版本，有 Bug)
if (ok) { onDone(); return; } // ← 未 await！
```

调用方（BlogCreate/BlogEdit）看到 `ensureAuth` 返回 → 以为操作完成 → 关闭保存遮罩。实际 API 调用还在跑，失败后无人处理。

**修复**: 彻底简化 `ensureAuth`——去掉所有 Promise 链条，改为最直接的"查密码 → 调回调"模式:
```javascript
const ensureAuth = useCallback((callback: () => void) => {
    const pw = api.getPassword();
    if (pw) {
      api.login(pw).then(ok => {
        if (ok) { callback(); return; }
        api.clearPassword();
        setWaiting(() => callback);
      }).catch(() => {
        api.clearPassword();
        setWaiting(() => callback);
      });
      return;
    }
    setWaiting(() => callback);
}, []);
```

三种路径:
- 密码存在且有效 → 验证通过 → `callback()` 同步执行
- 密码存在但失效 → 清空 → `setWaiting(callback)` → 显示 PasswordGate
- 密码不存在 → `setWaiting(callback)` → 显示 PasswordGate

**日期**: 2026-05-28
**涉及文件**: `src/hooks/useAuth.tsx`（重写 5 次以上）

### Bug 5: CDN 缓存导致 PUT 后 GET 返回旧数据

**现象**: 编辑保存成功，但刷新页面后内容变回旧值。

**根因**: Cloudflare CDN 对 `_worker.js` 的 GET 响应进行缓存。Cache-Control 头未在早期版本的 Worker 中设置。

**修复**: 四层防缓存:
1. **Worker 响应头**: 所有响应 `Cache-Control: no-cache, no-store, must-revalidate`
2. **客户端 fetch**: `cache: "no-store"` 参数
3. **URL 防缓存**: GET 请求 URL 加 `?_t=Date.now()`
4. **Service Worker**: `/api/*` 路由 `NetworkOnly` 策略

**日期**: 2026-05-20
**涉及文件**: `_worker.js`, `src/lib/api.ts`, `src/app/sw.ts`

---

### Bug 6: Service Worker 缓存 API 404

**现象**: F12 显示 "404 Not Found (来自 service worker)"——API 请求被 SW 拦截并返回缓存中的 404。

**根因**: Serwist 默认 `defaultCache` 包含 CacheFirst 策略，首次返回 404 后该响应被 SW 缓存，后续请求全部返回缓存 404，实际 API 请求永不发出。

**修复**: `src/app/sw.ts` 加自定义规则——`/api/*` 使用 `NetworkOnly` 策略:
```typescript
runtimeCaching: [
    ...defaultCache,
    {
        matcher: ({ url }) => url.pathname.startsWith("/api/"),
        handler: new NetworkOnly(),
    },
],
```

**日期**: 2026-05-28
**涉及文件**: `src/app/sw.ts`

### Bug 7: 中文 slug 在 Worker 中未被 URL 解码

**现象**: 含中文标题的文章（如"强强强强"）创建后 GET 返回 404，但 `/api/posts` 列表包含该文章。

**根因**: `_worker.js` 的 `param()` 函数直接做 `path.split("/")`。如果 Cloudflare Workers 的 `new URL(req.url).pathname` 在某些情况下未解码百分号编码，`param(3)` 返回的是 `%E5%BC%BA%E5%BC%BA` 而非 `强强`。数据库 slug 存储为 UTF-8 原文，WHERE slug = 编码串自然查不到。

**修复**: `param()` 强制 `decodeURIComponent`:
```javascript
const param = (n) => {
    const p = path.split("/")[n] || "";
    try { return decodeURIComponent(p); } catch { return p; }
};
```

**日期**: 2026-05-28
**涉及文件**: `_worker.js`

---

### Bug 8: 中文标题 slug 为空

**现象**: 旧文章 slug 为 ""（空字符串），导致无法通过 URL 访问。

**根因**: 早期 slug 生成逻辑 `body.title.toLowerCase().replace(/[^\w-]/g, "")` 中 `\w` 不匹配中文字符，中文标题全部被过滤→slug 为空→所有中文文章同 slug→ON CONFLICT 覆盖。

**修复**:
1. `makeSlug()` 改为保留中文字符: `replace(/[^a-z0-9一-鿿\-]/g, "")`
2. 加时间戳后缀: `+ "-" + Date.now().toString(36)` 确保唯一性
3. Worker 加 `/api/fix-slugs` 端点修复历史数据

**日期**: 2026-05-19
**涉及文件**: `_worker.js`

---

### Bug 9: Webpack JSX 解析 Bug（Next.js 14.2.35 特有问题）

**现象**: `npm run build` 报错 `Unexpected token div. Expected jsx identifier`——完全合法的 JSX 无法编译。

**根因**: Next.js 14.2.35 的 webpack JSX 解析器对多行 `return (<JSX>)` 存在 bug。触发条件:
- JSX 行 > ~250 字符
- `return (` 后 `<div>` 换行
- 函数体内有 `const` 声明后再 `const el = <div>...</div>`

**规避方法**:
1. 所有 JSX 赋值给变量: `const el = <div>...</div>; return el;`
2. 长属性拆成多行，每行 < 150 字符
3. 大组件拆成多个小文件（每个 < 100 行）
4. 博客页面拆为: `page.tsx`(10 行) → `BlogShell.tsx`(路由) → 4 个子组件

**日期**: 2026-05-28（多次被此问题阻塞构建）
**涉及文件**: 所有含 JSX 的组件

---

### Bug 10: Next.js Suspense 导致状态重置

**现象**: `router.push("/blog")` 后组件状态归零，posts 数组变为空。

**根因**: `<Suspense>` 包裹的组件在 `router.push` 同页导航时会重新挂载，所有 `useState` 重置。而 `router.push` 用于博客创建/编辑后跳转。

**修复**: 用 `window.location.href`（整页重载）替代 `router.push`（客户端导航）。虽牺牲了 SPA 平滑过渡，但彻底消除状态管理复杂度:
```javascript
const go = (url: string) => {
    window.location.href = url + (url.includes("?") ? "&" : "?") + "_t=" + Date.now();
};
```

配合模块级变量缓存，同 session 内重载后快速恢复:
```javascript
let gPosts: Post[] = [];
let gTags: TagStat[] = [];
```

**日期**: 2026-05-21
**涉及文件**: `src/app/blog/page.tsx`, `BlogShell.tsx`

## 四、最终代码架构

### 博客模块文件结构

```
src/app/blog/page.tsx              ← 10 行入口（Suspense 包裹）
src/components/blog/BlogShell.tsx  ← ~110 行路由层
    ├── 状态: posts, tags, html, loaded
    ├── 数据: useEffect 加载（仅首次）
    ├── 认证: ensureAuth → PasswordGate
    └── 路由分发:
        ├── ?new=1          → BlogCreate → Editor
        ├── ?edit=1&slug=x  → BlogEdit   → Editor
        ├── ?slug=x         → BlogDetail
        └── (无参数)         → BlogList + 搜索 + 侧边栏

src/components/blog/BlogCreate.tsx  ← 新建视图
src/components/blog/BlogEdit.tsx    ← 编辑视图
src/components/blog/BlogDetail.tsx  ← 详情视图（编辑/删除按钮）
src/components/blog/BlogList.tsx    ← 列表视图（搜索+标签+侧边栏）
src/components/blog/BlogContent.tsx ← Markdown 渲染 + 浮动 TOC
src/components/ui/Editor.tsx        ← 通用 Markdown 编辑器
src/components/ui/PasswordGate.tsx   ← 密码弹窗
src/hooks/useAuth.tsx               ← 全局认证（30 行，零 Promise 链）
src/lib/api.ts                      ← API 客户端（cache: no-store + _t）
_worker.js                          ← API 服务端（零依赖纯 JS，~250 行）
```

### 数据流

```
创建: Editor → onSaved(data) → BlogCreate → handleCreate → ensureAuth
       → api.savePost → Worker POST → INSERT ON CONFLICT → go("/blog?slug=xxx")
       → 整页重载 → React 加载 → GET /api/posts → 渲染列表

编辑: Editor → onSaved(data) → BlogEdit → handleUpdate → ensureAuth
       → api.updatePost → Worker PUT → UPDATE → go("/blog?slug=xxx")
       → 整页重载 → 渲染详情（含重试机制）

删除: BlogDetail → handleDelete → ensureAuth
       → api.deletePost → Worker DELETE → go("/blog")
       → 整页重载 → 渲染列表
```

### 认证流

```
ensureAuth(callback)
  ├─ 密码在 sessionStorage
  │   ├─ api.login 验证 → 通过 → callback()
  │   └─ api.login 验证 → 失败 → 清空 → PasswordGate → callback()
  └─ 密码不在 → PasswordGate → api.login → 存 sessionStorage → callback()
```

## 六、教训

1. **D1 主从延迟是设计特性不是 Bug**——分布式数据库有复制延迟，不要在主库写入后立即从库回读。前端本地状态是最可靠的数据来源。

2. **Cloudflare 网络环境特殊**——workers.dev 被墙、CDN 自动缓存、URL 编码不确定，每个环节都可能引入问题。

3. **Next.js 14.2.35 + `output: "export"` 限制多**——webpack JSX 解析有 bug、Suspense 导航会重置状态、`useSearchParams` 必须 Suspense 包裹。用 `window.location.href` 代替 `router.push` 是务实选择。

4. **不要过度设计**——`ensureAuth` 被重写了 5 次，最终版本只有 30 行。简单的"查密码 → 调回调"模式比复杂的 Promise 链更可靠。

5. **Service Worker 是双刃剑**——PWA 离线是好事，但 SW 缓存 API 响应会导致"修了代码但问题依旧"的假象。调试时先 Unregister SW。

6. **前端直接调 API 是最好的调试手段**——绕过所有框架层，用 `fetch` 在 Console 直接测试，立即可知问题在前端还是后端。
